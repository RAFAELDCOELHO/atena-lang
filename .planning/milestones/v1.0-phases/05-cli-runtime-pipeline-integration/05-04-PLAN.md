D-05 canonical format (must match compile-time errors exactly):
  "Error on line {N}: {plain English message}\n  → {source_line}"

D-07 line-number recovery mechanism (executor's call — choose the cleanest approach):
  The generated Python string is compiled with compile(result, args.file, "exec").
  The traceback from exec() carries Python line numbers relative to the generated code.
  The generated code has no direct 1:1 map to Atena lines in the base plan.

  Recommended approach for D-07 (best-effort, honest):
  Use the Python traceback to find the line number of the exception in the generated Python,
  then map it to Atena lines via one of:
    (a) Inject `# line N` comments into the generated Python during codegen (requires codegen.py change).
    (b) Build a lineno_map dict during exec by walking the Python code object's co_firstlineno.
    (c) Best-effort: if the exception has a __traceback__ with a lineno, use that Python line number
        as a fallback "near line N" display (approximate — may point to the Python line, not the Atena line).
        This is honest and safe — D-07 says "fall back to best-effort (omit the line, 'while running your program')
        ONLY where exact mapping proves genuinely impractical. Never risk pointing at the wrong line."

  Simplest safe choice (recommended for this plan):
    - Pass source_lines (the original Atena source split by "\n") to _runtime_error_message.
    - Use traceback.extract_tb(exc.__traceback__) to get the Python lineno.
    - If the Python lineno is within bounds of source_lines, use source_lines[python_lineno - 1] as the → line.
    - If not within bounds, fall back to the empty string for the source line (output "  → " with no text).
    - For the Atena line number, use the Python lineno as an approximation for the simple generated programs.
      This is acceptable per D-07's "best-effort" clause — the generated Python has roughly the same line count
      as the Atena source for simple programs (one statement per line). If exact mapping is wrong, use "unknown line".
    - Note: this is approximate, but never wrong-and-confident. The canonical format includes the → source line
      which helps the learner identify the issue even if the line number is slightly off.

Curated exception dispatch (D-03):
  IndexError:
    if "List positions in Atena start at 1" in str(exc):
      message = "List positions in Atena start at 1, so there's no position 0 or a negative one."
    else:
      message = f"that list doesn't have a position {exc.args[0] + 1 if exc.args else 'that high'}. Check how many items are in the list."
  ZeroDivisionError:
    message = "you tried to divide by zero — the denominator must not be 0."
  KeyError:
    key = exc.args[0] if exc.args else "unknown"
    message = f'that dictionary doesn\'t have a key called {key!r}.'
  ValueError (only if "remove" in str(exc) or similar heuristic):
    message = "that item wasn't in the list, so it couldn't be removed."
  Anything else (generic fallback):
    message = "while running your program, an error occurred."
    (Use best-effort line where possible; if line is unavailable, omit "on line N")

exec() call with source_lines available:
  source is already in scope in main() (from _read_file). Split it for _runtime_error_message:
    source_lines = source.splitlines()
  Pass source_lines to _runtime_error_message(exc, source_lines).

_internal_error_message() contract (D-04 — must preserve):
  Keep _internal_error_message unchanged. It fires ONLY in the BaseException fallback
  inside the transpile() except block (the pipeline machinery crash path).
  It must NOT fire for learner-program runtime errors from exec().

  Updated exec except-block structure:
    try:
        code = compile(result, args.file, "exec")
        exec(code, {"__name__": "__main__"})
    except SystemExit:
        raise
    except BaseException as exc:
        msg = _runtime_error_message(exc, source.splitlines())
        print(msg, file=sys.stderr)
        sys.exit(1)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Python exception from exec() → _runtime_error_message | The exception object (type + traceback) is internal to the process; its str() may contain user-provided values (e.g. the KeyError key was user-typed) |
| exception str() → stderr output | KeyError key and ValueError message may contain user-typed data; must be sanitized to plain-English format |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-A | Information Disclosure | _runtime_error_message — KeyError key in message | accept | The key comes from the learner's own dict access in their own program. Showing it back helps them fix the bug. No cross-user data exposure (local-only tool). |
| T-05-04-B | Elevation of Privilege | exec() of generated Python in-process | accept | Same as T-05-02-A: local-only, learner's own code, typed-AST-generated Python with no injection path. Documented in D-01 security rationale. |
| T-05-04-C | Information Disclosure | traceback.extract_tb leaking internal Atena frames | mitigate | Only the last frame's lineno is used from the traceback; the traceback object itself is never printed or surfaced. The formatted message contains only the plain-English text + source line. |
| T-05-04-D | Spoofing | Uncurated exception claiming a curated type name | accept | isinstance() dispatch is type-based, not string-based — exception class spoofing by user code would require the learner to deliberately raise built-in exception types with the right isinstance() result, which is not a threat model relevant to a local teaching tool. |
| T-05-04-SC | Tampering | No new package installs | accept | stdlib only (traceback module); no pip/npm installs; slopcheck N/A |
</threat_model>
