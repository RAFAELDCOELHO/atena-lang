<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → cli._read_file | The learner-supplied .atena file path is read from the filesystem; path comes from CLI args |
| internal exception → stderr | Any uncaught Python exception must NOT cross this boundary as a traceback |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-05-01 | Information Disclosure | uncaught exception → stderr (D-02 / Pitfall 6) | mitigate | Outer `except BaseException` in main() converts ALL unexpected exceptions to the blame-free internal-error message; `SystemExit` and `KeyboardInterrupt` are explicitly re-raised before the catch-all. Tests C-7 and C-8 assert no traceback text in output. |
| T-00-05-02 | Tampering | file path from CLI arg | accept | The learner runs `atena run ./their-file.atena` on their own machine; no server context; path traversal risk is negligible for a local offline tool. os.path.basename is used in error messages (not full path) to avoid leaking directory structure. |
| T-00-05-03 | Denial of Service | reading a very large .atena file | accept | Teaching programs are small (hundreds of lines at most); no file size cap needed. If pathological input were a concern, a size check would be added here. |
| T-00-05-04 | Information Disclosure | full file path in error message | mitigate | Error messages use only the filename (basename), not the full path (e.g., "I couldn't find a file called \"missing.atena\"." not the full /home/user/... path). This matches the CONTEXT.md exemplar. |
</threat_model>
