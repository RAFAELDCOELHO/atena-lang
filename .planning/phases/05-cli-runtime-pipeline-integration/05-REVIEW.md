## Summary

Phase 5 wires the four-phase pipeline under the `atena run` / `atena build` CLI and
adds runtime-error translation so no Python traceback reaches the learner. The
no-traceback guarantee mostly holds (every escape path I probed was caught), and the
file-I/O error messaging is solid.

However, the **runtime error line-number mapping is fundamentally broken** and the test
suite does not catch it. `_runtime_error_message()` takes the Python traceback's line
number and indexes it directly into the *Atena* source lines — but the generated Python
is NOT line-aligned with the Atena source. Codegen prepends a `# Generated by Atena`
header (constant +1 shift), optionally prepends multi-line helper preambles
(`_atena_index`, `_atena_concat` — variable shift), and for errors raised inside
functions or injected helpers the traceback's last frame points at the *helper/def body*,
not the call site. The result: the "Error on line N" number is wrong in essentially every
real program, and the `→ <source line>` is usually **empty**. This directly violates the
core project invariant ("name the line and show the offending source line").

All 37 phase tests pass because the runtime-error tests monkeypatch `transpile()` to
return 1-line snippets with no header and no preamble, so the broken offset coincidentally
lands in-bounds. They assert only that the strings `"Error on line"` and `"→"` appear —
never that the number or the source line is correct.

Two more correctness defects compound this: a `ValueError` catch-all that mislabels every
`ValueError` as "that item wasn't in the list", and a `compile()`-time `SyntaxError`
(an internal codegen bug) being misclassified as a learner runtime error. Plus a data-loss
path in `build` (no extension check → can overwrite the input file).

## Critical Issues

### CR-01: Runtime error line numbers are wrong; offending source line is usually empty

**File:** `src/atena/cli.py:113-155` (`_runtime_error_message`), root cause spans `src/atena/codegen.py:165-202`

**Issue:** `_runtime_error_message` extracts `python_lineno` from the traceback and then
maps it directly onto `source_lines` (the *Atena* source) at line 132-136:

```python
python_lineno = tb_frames[-1].lineno if tb_frames else None
if python_lineno is not None and 1 <= python_lineno <= len(source_lines):
    source_line = source_lines[python_lineno - 1].strip()
line_display = python_lineno if python_lineno is not None else "unknown"
```

The generated Python is not line-aligned with the Atena source. Codegen
(`codegen.py:200-202`) prepends `# Generated by Atena` (+1 line), and
(`codegen.py:208-215`) optionally prepends `_atena_index` (4 lines) and/or
`_atena_concat` (2 lines) helper bodies. `ast.fix_missing_locations` assigns line numbers
from AST structure, not from any preserved Atena `lineno`. Proven behavior:

| Atena program | Error at Atena line | Reported "Error on line" | `→` source line |
|---|---|---|---|
| `show 1` / `show 2` / `x = 10 / 0` | 3 | **4** | *(empty)* |
| `nums=[1,2,3]` / `show nums[10]` | 2 | **3** | *(empty)* |
| concat + index helpers, error on line 2 | 2 | **5** | *(empty)* |
| `function divide...` / `show divide(10,0)` | 4 | **3** (points at `return a/b` def body) | *(empty)* |
| computed neg index → `_atena_index` raise | 3 | **4** (inside injected helper) | *(empty)* |

The line number is wrong in every multi-line program, and because the wrong number
usually exceeds `len(source_lines)`, the `1 <= python_lineno <= len(source_lines)` guard
fails and `source_line` stays `""` — so the learner sees `→` followed by nothing. This is
the exact opposite of the invariant: "name the line and show the offending source line."

**Fix:** The Python traceback lineno cannot be trusted for Atena line mapping. Carry the
Atena `lineno` through codegen onto the emitted `ast` nodes (the Atena AST nodes already
carry `line`/`col`; set `node.lineno`/`col_offset` on the corresponding Python `ast` nodes
instead of relying on `fix_missing_locations`), then resolve the *call-site* Atena line
from the appropriate traceback frame. Minimally, build a Python-line → Atena-line map at
codegen time and pass it into `_runtime_error_message` instead of indexing `source_lines`
directly. Until a real mapping exists, do not print a specific line number you cannot
prove correct — print the message without a fabricated `line N` rather than a wrong one.

### CR-02: `ValueError` catch-all mislabels every ValueError as a failed list removal

**File:** `src/atena/cli.py:144-145`

**Issue:** Any `ValueError` is hard-coded to the list-removal message:

```python
elif isinstance(exc, ValueError):
    message = "that item wasn't in the list, so it couldn't be removed."
```

`ValueError` is raised by far more than `list.remove`. Proven: a learner program that
runs `int("notanumber")` (or any future numeric-conversion / `list.index` path) produces
`Error on line 1: that item wasn't in the list, so it couldn't be removed.` with a `→`
line that is unrelated to the actual fault. The message is confidently, specifically
wrong — worse for a teaching tool than a generic message, because it sends the learner
hunting for a list-removal bug that doesn't exist.

**Fix:** Narrow the `ValueError` branch to the case it actually describes (inspect
`str(exc)` for the `list.remove(x): x not in list` signature, mirroring the `IndexError`
branch at line 147), and route all other `ValueError`s to the generic
"while running your program, an error occurred." fallback:

```python
elif isinstance(exc, ValueError):
    if "not in list" in str(exc):
        message = "that item wasn't in the list, so it couldn't be removed."
    else:
        message = "while running your program, an error occurred."
```

### CR-03: Generated-code `SyntaxError` at `compile()` is misclassified as a learner runtime error

**File:** `src/atena/cli.py:203-212`

**Issue:** `compile(result, args.file, "exec")` sits inside the same `try` as `exec`, and a
failure there is caught by `except BaseException` and sent to `_runtime_error_message`.
But a `SyntaxError` from `compile()` means codegen emitted invalid Python — an **internal
Atena bug**, not a learner error. The code's own docstrings (D-04 split, `cli.py:120-124`
and `pipeline.py:13-15`) state internal bugs must use the "Something went wrong inside
Atena — this isn't your fault" wording. Proven: feeding invalid generated code produces
`Error on line 204: while running your program, an error occurred.` — wrong category,
plus a nonsense line number (204 is a frame from `compile` internals). Note codegen runs
a GEN-05 `ast.parse` self-check (`codegen.py:175`), so in practice this should not fire —
but if it ever does, the CLI blames the learner for a transpiler bug.

**Fix:** Move `compile()` out of the runtime `try`, or catch `SyntaxError` from `compile`
separately and route it to `_internal_error_message`:

```python
try:
    code = compile(result, args.file, "exec")
except SyntaxError as exc:               # codegen emitted invalid Python -> internal bug
    print(_internal_error_message(exc), file=sys.stderr)
    sys.exit(1)
try:
    exec(code, {"__name__": "__main__"})
except SystemExit:
    raise
except BaseException as exc:             # genuine learner runtime error
    print(_runtime_error_message(exc, source.splitlines()), file=sys.stderr)
    sys.exit(1)
```

## Warnings

### WR-01: `build` can overwrite the input file (data loss) — no extension validation

**File:** `src/atena/cli.py:190-191`

**Issue:** `out_path = os.path.splitext(args.file)[0] + ".py"` with no check that the input
ends in `.atena`. `atena build foo.py` yields `out_path == "foo.py"` — the same file —
silently overwriting the user's input with generated code. Any input whose stem + `.py`
equals the input path is destroyed. There is no extension guard anywhere in `cli.py`.

**Fix:** Validate the input extension and/or refuse to write when `out_path == args.file`:

```python
out_path = os.path.splitext(args.file)[0] + ".py"
if os.path.abspath(out_path) == os.path.abspath(args.file):
    print(f'I can only build files ending in ".atena".', file=sys.stderr)
    sys.exit(1)
```

(Optionally reject non-`.atena` inputs outright in both `run` and `build`.)

### WR-02: Phase tests assert error *format* but never assert line-number or source-line correctness

**File:** `tests/test_cli.py:520-565` (C-14), `:572-608` (C-20), `:615-654` (C-21), `:661-697` (C-22), `:704-740` (C-23)

**Issue:** These tests monkeypatch `transpile` to return 1-line Python snippets with no
header comment and no helper preamble (e.g. C-14 returns `"x = 1 / 0\n"`). That bypasses
the real codegen, so the broken offset in CR-01 happens to land in-bounds. They assert
only that `"Error on line"` and `"→"` appear — never the actual number, never that the
source line after `→` is non-empty or correct. As a result the suite is fully green while
CR-01 ships. Tests give false confidence here.

**Fix:** Add at least one end-to-end test that runs the *real* pipeline on a multi-line
`.atena` program whose runtime error is on a known line (e.g. line 3), and assert the
reported line equals 3 AND the `→` line equals the trimmed Atena source of line 3. That
test fails today and would have caught CR-01.

### WR-03: `KeyboardInterrupt` during a learner program is swallowed and mislabeled

**File:** `src/atena/cli.py:206-212`

**Issue:** The `exec` handler re-raises only `SystemExit` (line 206-207); `except
BaseException` (line 208) therefore catches `KeyboardInterrupt`. Proven: a learner
pressing Ctrl-C mid-run gets `Error on line N: while running your program, an error
occurred.` instead of a clean interrupt. No traceback leaks (good), but a deliberate user
interrupt should not be reported as a program error.

**Fix:** Re-raise `KeyboardInterrupt` alongside `SystemExit` (or exit silently on it):

```python
except (SystemExit, KeyboardInterrupt):
    raise
except BaseException as exc:
    print(_runtime_error_message(exc, source.splitlines()), file=sys.stderr)
    sys.exit(1)
```

### WR-04: Repeated `print(errors.report(), ...)` + `return None` block in `transpile`

**File:** `src/atena/pipeline.py:52-66`

**Issue:** The identical three-line gate (`if not errors.is_empty(): print(errors.report(),
file=sys.stderr); return None`) is duplicated after each of the three phases. Beyond
duplication, this couples the pipeline to stderr printing — a caller that wants the report
as a string (e.g. to show it in an editor, a web playground, or to test it) cannot get it
without capturing stderr. `test_pipeline.py:99-106` already has to use `capsys` for this
reason.

**Fix:** Factor the gate into a small helper, and consider returning/raising a structured
result so the caller decides where the report goes:

```python
def _gate(errors) -> bool:
    if not errors.is_empty():
        print(errors.report(), file=sys.stderr)
        return True
    return False
...
tokens = Lexer(source, errors).tokenize()
if _gate(errors):
    return None
```

### WR-05: `compile()` uses the full input path as the code filename

**File:** `src/atena/cli.py:204`

**Issue:** `compile(result, args.file, "exec")` embeds the full filesystem path into the
compiled code object. Every other user-facing message in this module deliberately uses
`os.path.basename` (see `_file_error_message`, `cli.py:79-91`). If any future change lets a
frame surface (or a learner's program itself raises and prints), the absolute path leaks.
It is also inconsistent with the basename-only convention.

**Fix:** Use the basename as the compile filename: `compile(result,
os.path.basename(args.file), "exec")`.

## Info

### IN-01: Redundant `except ... : raise` chain in `_read_file`

**File:** `src/atena/cli.py:60-73`

**Issue:** The four `except FileNotFoundError/IsADirectoryError/PermissionError/
UnicodeDecodeError: raise` clauses (lines 63-70) are no-ops — they catch and immediately
re-raise the same exception type, which is identical to not catching it. Only the final
`except OSError` → `PermissionError` remap is load-bearing. The dead clauses add noise and
imply intent that isn't there.

**Fix:** Delete the four pass-through `except` clauses; keep only the `OSError` remap:

```python
try:
    with open(path, encoding="utf-8") as fh:
        return fh.read()
except (FileNotFoundError, IsADirectoryError, PermissionError, UnicodeDecodeError):
    raise
except OSError as exc:
    raise PermissionError(str(exc)) from exc
```

### IN-02: `filename` parameter threaded through `transpile` but never used

**File:** `src/atena/pipeline.py:29-69`

**Issue:** `transpile(source, filename)` accepts `filename` and the docstring says it is
"reserved for future use", but it is never referenced in the body. The errors come pre-
formatted from `ErrorCollector`, which has no filename context. This is acceptable as a
forward-looking signature, but note that the line-mapping problem in CR-01 is precisely
the "future use" this param hints at — wiring source/line provenance through the pipeline
would likely need it.

**Fix:** Either use it (pass to codegen for a line map / into the report) or drop it until
needed. No change required for correctness.

### IN-03: `_runtime_error_message` line-display type union is a symptom, not a feature

**File:** `src/atena/cli.py:136`

**Issue:** `line_display: int | str = python_lineno if python_lineno is not None else
"unknown"` produces messages like `Error on line unknown: ...`. Given CR-01, "unknown" is
arguably more honest than a confidently-wrong integer — but the mixed `int | str` and the
`"unknown"` sentinel are a sign the line-number provenance is missing upstream. Once a real
Atena line map exists (CR-01 fix), this should resolve to a concrete Atena line or be
omitted entirely.

**Fix:** Tie resolution to the CR-01 fix; avoid emitting `line unknown` to learners — if no
reliable line is available, phrase the message without a line clause.

## Resolution (2026-06-14 gap closure)

Root-cause fix for CR-01: the `atena run` path now compiles the Python **AST
module** (whose nodes carry the originating Atena `lineno`, stamped in codegen)
instead of the unparsed source string. CPython's line table then maps runtime
tracebacks straight back to the learner's Atena line — no offset math, no empty
`→`. New `pipeline.compile_for_run()` does this; `pipeline.transpile()` is
unchanged for `build`.
