<!-- Key details for this plan's edge cases -->

keyword.kwlist (Python 3.11) includes: False, None, True, and, as, assert, async, await,
  break, class, continue, def, del, elif, else, except, finally, for, from, global, if,
  import, in, is, lambda, nonlocal, not, or, pass, raise, return, try, while, with, yield
  (35 keywords — all become trailing-underscore when used as Atena identifiers)

Note: "and", "or", "not", "if", "else", "while", "return", "true", "false" are Atena
  keywords that the Atena lexer classifies as KEYWORD tokens (not IDENTIFIER tokens) — a
  learner cannot use them as variable names at all (the parser never produces Identifier nodes
  with those values). So in practice, the mangling only activates for Python keywords that
  Atena's lexer does NOT classify as keywords: class, import, lambda, yield, del, pass, etc.
  The mangle is still correct to apply universally (keyword.kwlist is the source of truth).

Nested repeat loop-var scheme (monotonic counter):
  First repeat → self._loop_counter starts at 0, uses "_atena_i0", increments to 1
  Inner repeat → uses "_atena_i1", increments to 2
  After inner body emitted → counter stays at 2 (MONOTONIC — never decremented)
  Outer body finishes → outer's loop var is already "_atena_i0", counter stays at 2
  Next sibling repeat → uses "_atena_i2"
  This ensures every repeat across the whole program gets a unique loop var name.
  (The PATTERNS.md showed a decrement pattern — the monotonic approach is strictly safer
  and simpler; the executor should use monotonic.)

Verbatim-emission discipline checks:
  1. No re-str() wrap: source "x = 5\ny = 3\nshow x + y\n" — after analysis, both are
     NumberLiteral-typed, BinOp("+", Identifier("x"), Identifier("y")) stays as-is (no_coerce).
     Generated Python contains "x + y" NOT "str(x) + y" or "str(x) + str(y)".
  2. No re-shift: source "grades = [5, 7, 9]\nshow grades[2]\n" — analyzer folds index 2→1.
     Generated Python contains "grades[1]" NOT "grades[0]" or "grades[2-1]".
  3. _atena_concat path: source "name = ask 'Enter: '\nshow name + 1\n" — ask result is
     typed str, so str + number → static coerce_right: generates "show name + str(1)".
     But if "x + y" where both are function params (unknown type): generates
     "_atena_concat(x, y)".

_atena_index runtime behavior:
  i=0 → raises IndexError("List positions in Atena start at 1.") — plain English
  i=1 → returns 0 (correct: 1-based → 0-based)
  i=-1 → raises IndexError (i < 1 check catches negatives too)
