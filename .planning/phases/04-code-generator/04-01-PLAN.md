From src/atena/parser.py (key structural insight for dot-write):
  _parse_assignment() handles IDENTIFIER ASSIGN → produces Assign(name=str, value=Node)
  _parse_expression_statement() rejects non-FunctionCall bare expressions
  The fix needed: when _dispatch_statement sees IDENTIFIER followed by DOT (not ASSIGN),
  parse the full LHS postfix chain (becomes DotAccess), then expect ASSIGN, then parse RHS.
  Produce: Assign(name="__dot__", value=...) is NOT the right shape.
  Correct shape: a new node type would be ideal but adds to ast_nodes.py.
  Simpler: extend Assign to support a DotAccess target — use a sentinel name like
  the string representation of the dot path, OR use a new `DotAssign` node.
  DECISION: Extend _parse_statement to detect IDENTIFIER DOT FIELD ASSIGN and produce
  Assign(name=f"{obj}.{field}", value=rhs) is fragile. Best: parse as Assign where
  the `name` field is not used for dot-write; instead store a `dot_target: DotAccess | None`
  attribute on Assign — but ast_nodes.py is read-only contract (no changes).
  SIMPLEST COMPATIBLE: use Assign(name="", value=rhs) plus a new sibling node
  DotAssign is not in ast_nodes.py scope.
  ACTUAL CONTRACT (from PATTERNS.md line 221-229): "parser produces an Assign where
  node.name is a synthetic compound". Use Assign(name="__dot__{obj}.{field}", value=rhs)
  as a sentinel format the codegen can detect and parse — OR keep DotAccess as
  an expression in a special Assign wrapper.
  BEST APPROACH: Parse dot-write as Assign(name="", value=rhs) with a `_dot_target`
  field set dynamically on the Assign object. Python dataclasses are mutable; add
  `_dot_target: DotAccess | None = None` dynamically (not via ast_nodes.py).
  CLEANEST: After Parser fix: Assign node gets a dynamically-set attribute
  `node._dot_target = DotAccess(...)` so codegen checks `hasattr(node, '_dot_target')`.
  Executor: pick the cleanest approach consistent with "never modify ast_nodes.py"
  and "codegen sees the Assign and emits a subscript Store".

    The current handler for IDENTIFIER-led statements only dispatches to `_parse_assignment()` when `self._peek().type == TokenType.ASSIGN`. When the IDENTIFIER is followed by DOT (e.g., `student.grade = 10`), it falls through to `_parse_expression_statement()` which rejects the non-FunctionCall expression.

    Add a dot-write branch BEFORE the existing IDENTIFIER + ASSIGN check. When the current token is IDENTIFIER and `self._peek().type == TokenType.DOT`, parse the full LHS as a postfix expression via `_parse_postfix(_parse_primary())` to get a DotAccess node, then expect ASSIGN (`=`), then parse the RHS expression.

    Produce the result as a standard `Assign` node with `name=""` and the DotAccess node stored on a dynamically-set attribute `_dot_target` on the Assign instance. This keeps `ast_nodes.py` unchanged (no new node types) while giving codegen a clean signal:
    - `node.name == "" and hasattr(node, "_dot_target")` → dot-write assignment
    - `node._dot_target` is a `DotAccess(target=Identifier("student"), name="grade")`

    Restrictions:
    - Do NOT modify `ast_nodes.py` — the Assign dataclass is a locked contract.
    - Do NOT use fragile string encoding like `"student.grade"` in `name`.
    - Do NOT attempt to handle nested dot-write chains beyond one level (student.grade.subfield) — v1.0 scope is flat dict access only.
    - Use `_end_statement()` after parsing the RHS (same as `_parse_assignment` does).
    - Set `line` and `source_line` from the first IDENTIFIER token, same as `_parse_assignment`.
